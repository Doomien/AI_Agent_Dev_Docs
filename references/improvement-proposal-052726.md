# Improvement Recommendations — 2026-05-27

**Source:** GitHub Copilot Chat session  
**Status:** Unreviewed — for consideration  
**Date:** 2026-05-27

---

## Overview

The following recommendations were generated by reviewing the current state of `Doomien/AI_Agent_Dev_Docs`. They are organized by estimated impact to an agent developer adopting or working within this framework.

---

## 🔴 High Impact

### 1. Add `.github/copilot-instructions.md`

`AGENTS.md` is well-suited for OpenAI-style agents (Codex, Devin, etc.), but GitHub Copilot specifically reads `.github/copilot-instructions.md` as its cold-start file. Adding a parallel file there — even a thin wrapper that points to `AGENTS.md` — would make the repo immediately usable with Copilot without any manual setup.

**Action:** Create `.github/copilot-instructions.md` with repo context, key rules, and load-on-demand links mirroring `AGENTS.md`.

---

### 2. Enable Template Repository

The `templates/` directory is described as something to "copy into a new project's `docs/`", but the repo itself is not configured as a [GitHub template repository](https://docs.github.com/en/repositories/creating-and-managing-repositories/creating-a-template-repository). Enabling this setting adds a **"Use this template"** button, letting developers scaffold a new project with the framework already in place.

**Action:** Go to **Settings → General → Template repository** and enable the checkbox.

---

### 3. Add `CHANGELOG.md`

The repo has versioned material in `archive/` (v01, v02, v03) and a pending improvement proposal (`references/improvement-proposal-052526.md`). A changelog would help agent developers — and humans — understand what changed between framework versions, which is critical when multiple projects have adopted the framework at different points in time.

**Action:** Create `CHANGELOG.md` at the repo root. Backfill entries for v01 → v02 → v03 (current `templates/`) transitions using the archive as a reference.

---

## 🟡 Medium Impact

### 4. Add GitHub Issue Templates

Agents are expected to open PRs, so structured issue templates would give them (and humans) a consistent intake format aligned with the existing workflow.

**Suggested templates in `.github/ISSUE_TEMPLATE/`:**
- `doc-improvement.md`
- `new-guide-request.md`
- `broken-link-report.md`

**Action:** Create `.github/ISSUE_TEMPLATE/` with the above files.

---

### 5. Add a PR Template

`guides/contributing.md` already contains a PR checklist. Surfacing it automatically as `.github/pull_request_template.md` ensures agents and contributors always get the checklist pre-filled when opening a PR, without needing to read the contributing guide first.

**Action:** Extract the PR checklist from `guides/contributing.md` into `.github/pull_request_template.md`.

---

### 6. Add Repository Topics

The repo currently has **no topics set**, limiting discoverability. Adding relevant tags would help other agent developers find the framework.

**Suggested topics:** `ai-agents`, `llm`, `developer-docs`, `prompt-engineering`, `copilot`, `workflow`, `templates`

**Action:** Go to the repo homepage → gear icon next to "About" → add topics.

---

### 7. Add a License

No license is currently set. Without one, the repository is technically all-rights-reserved by default, which prevents other developers from legally adopting or adapting the framework — contrary to its apparent intent.

**Action:** Add a `LICENSE` file. `MIT` or `CC BY 4.0` (for a docs-only repo) are common choices.

---

## 🟢 Lower Impact / Polish

### 8. Add a `GLOSSARY.md` or Inline Term Definitions

Terms like "PM/IC/QA loop," "compaction," and "cadence" appear throughout the framework but are not defined in a single authoritative location. A glossary would reduce ambiguity for new agents and developers encountering the framework for the first time.

**Action:** Create `GLOSSARY.md` at the repo root, or add a "Terms" section to `AGENTS.md`.

---

### 9. Add a Broken-Link CI Check

`AGENTS.md` explicitly notes that "cross-references between docs are load-bearing." A GitHub Actions workflow using a tool like [`lychee`](https://github.com/lycheeverse/lychee-action) would automatically catch broken internal links on every PR, removing the need for manual grep-based validation.

**Action:** Create `.github/workflows/link-check.yml` using the `lycheeverse/lychee-action`.

---

### 10. Add a Sample / Example Project

The framework is described but not demonstrated with a concrete example. A minimal filled-in sample — showing a completed `CONTEXT.md`, `PROGRESS.md`, and `HANDOFF.md` for a fictional project — would significantly reduce the time-to-first-use for a developer adopting this framework.

**Action:** Create `examples/sample-project/` containing filled-in versions of the core templates.
